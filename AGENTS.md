## Project Overview

Notionに保存されたフラッシュカードデータをAnkiへ出力する。

## Security Guideline of Code

コミット前に必ず以下の点を確認すること。

- 機密情報（パスワードやAPIキー、トークン）がハードコードされていないか
- すべてのユーザーの入力に対してバリデーションをしているか
- SQLインジェクションを防ぐようにしているか
- XSSを防ぐようにしているか
- CSRFへの対策は有効か
- エラーメッセージに機密情報を含んでいないか

機密情報のマネジメント: 機密情報を決してハードコードしないこと。環境変数またはシークレットマネージャーを使用すること。機密情報が流出した場合は即座に情報を変更すること。

セキュリティの問題が発覚した場合: 開発をストップする → セキュリティレビュー用のエージェントを使用する → クリティカルな問題を集成する → 機密情報を変更する → コードベース全体で同様の問題がないかをレビューする

## Coding Style

イミュータブル: 常に新しいオブジェクトを作成し、既存のオブジェクトを変更しないこと。

ファイル構成: 少数の大規模なファイルではなく、多数の小さなファイルを使用する。通常は200〜400行で最大800行とする。ファイルは機能/ドメインごとに整理し、凝縮度が高く、疎結合とすること。

バリデーション: すべてのユーザー入力に対してバリデーションチェックを行うこと。 バリデーション結果のメッセージは明確にすること。

コード品質のチェックリスト:

- 1つの関数は50行以下で、1つのファイルの量が200〜400行（最大800行）になっているか
- 4つ以上のネストをしていないか
- エラーハンドリングは適切か
- 機密情報をハードコードしていないか
- 関数や変数の命名は明瞭になっているか

## テスト要件

最小カバー率: 80%

テストの種類（すべて必須）:

- ユニットテスト: 個々の関数、ユーティリティー
- 結合テスト: APIエンドポイント

TDDワークフロー（必須）:

1. 最初にテストを書く（赤）: テストは失敗する必要がある
2. 最小限の実装を書く（緑）: テストは合格する
3. リファクタリング（改善）: カバレッジ率80％以上を検証

失敗時のトラブルシューティング: テストの分離をチェック → モックを検証 → 実装を修正

## 開発ワークフロー

1. 計画: プランナーエージェントを使用し、依存関係をリスクを特定し、フェーズに分割する
2. TDD: tdd-guideエージェントを使用し、最初にテストを記述し、実装し、リファクタリングする
3. レビュー: コードレビューエージェントをすぐに使用し、CRITICAL/HIGHの問題に対処する
4. コミット: 従来のコミット形式、包括的なPRサマリー

## Gitワークフロー

コミット形式: `<type>: <description>` - type: feat, fix, refactor, docs, test, chore, pref, ci

PRワークフロー: 完全なコミット履歴を分断 → 包括的な概要の下書きを作成 → テストを含める → `-u`フラグ付きでプッシュ

## パフォーマンス

コンテキスト管理: 大規模なリファクタリングや複数ファイルに対する機能追加では、コンテキストウィンドウの最後の20％の使用を避ける。単一の編集やドキュメント、簡単な修正などでは機密性の低いタスクより高い使用率を許容する。

## 成功指標

- すべてのテストが80％以上のカバレッジで合格していること
- セキュリティ上の脆弱性がないこと
- コードが可動性に優れ、保守しやすいこと
- パフォーマンスが許容範囲内であること
- ユーザー要件を満たしていること
